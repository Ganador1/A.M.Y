### 2.2 Esquema Propuesto `hypotheses` (SQL / Alembic)
Campos nuevos (añadir a tabla existente):

- `cross_validation_matrix` (JSON TEXT) → Matriz NxN con celdas `{compatibility: float, rationale: str, last_updated}`.

### 3.3 State Machine del Hypothesis Core
Estados mínimos (Fase 1): `DRAFT -> EXPANDED -> VALIDATED -> REFINED -> PUBLISHED -> ARCHIVED`.
Transiciones con guards:
- `expand`: requiere `domains_involved` detectados.
- `validate`: requiere outputs de agentes de dominio + risk gate.
- `refine`: se activa si `consensus_score < threshold` o conflictos.
- `publish`: exige `reproducibility_score >= min`, `risk_score <= max`.

```
async def resilient_run(adapter, payload):

        except TransientError: retry
        except FatalError: open_circuit(adapter)
```
Circuit breaker simple en memoria (contador fallos ventana).

## 4. Diseño de Domain Agents y Tool Adapters

### 4.1 Principios
- Separación: Agent (lógica científica + orquestación) vs Adapter (invocación herramienta concreta).
- Adaptadores intercambiables con fallback jerárquico.
- Observabilidad unificada (mismo contrato de métricas).

### 4.2 Interfaz Base `ToolAdapter`
```
class ToolAdapter(Protocol):
    name: str
    version: str
    timeout: int
    cost_estimate: float
    async def available(self) -> bool: ...
    async def run(self, payload: dict) -> dict: ...
    def describe(self) -> dict: ...
```
Wrapper estandariza salida:
```
{
  "status": "ok|error",
  "started_at": ts,
  "ended_at": ts,
  "metrics": {...},
  "artifacts": [{id,type,uri,hash}],
  "data": {...},
  "errors": []
}
```

### 4.3 Registro Dinámico
```
adapter_registry = {}

def register_adapter(domain: str, adapter: ToolAdapter, priority: int):
    adapter_registry.setdefault(domain, []).append((priority, adapter))
    adapter_registry[domain].sort(key=lambda x: x[0])
```
Selección: probar en orden prioridad; si `available()==False` o circuito abierto, pasar al siguiente.

### 4.4 Ejemplo Cadena Proteica (Pipeline)
1. `ProteinDesignAdapter` (ProtGPT2 / diffusion futuro)
2. `StructurePredictionAdapter` (AlphaFold3 → fallback ESMFold)
3. `DockingAdapter` (Mock DiffDock → real)
4. `StabilityHeuristicsAdapter` (DDG estimado heurístico)

Agente compone:
```
class ProteinDomainAgent(DomainAgent):
    async def evaluate(self, hypothesis: Hypothesis):
        design = await run_adapter_seq("protein.design", payload)
        structure = await run_adapter_seq("protein.structure", {seq: design.data.seq})
        docking = await run_adapter_seq("protein.docking", {pdb: structure.artifacts[0].uri})
        stability = await run_adapter_seq("protein.stability", {pdb: structure.artifacts[0].uri})
        return aggregate_metrics([...])
```

### 4.5 Métricas Estándar de Adapters
- `runtime_seconds`
- `energy_cost_estimate` (proxy: runtime * power_draw)
- `novelty_score` (cuando aplica, e.g. diseño)
- `confidence_score`
- `error_rate` (rolling)

### 4.6 Gestión de Errores
Clasificación:
- `TransientError` (retry backoff)
- `CapacityError` (requeue + degrade priority)
- `FatalError` (marcar circuito y fallback)

### 4.7 Instrumentación
Decorador `@adapter_metrics` envía:
- Counter `adapter_runs_total{adapter}`
- Histogram `adapter_latency_seconds{adapter}`
- Gauge `adapter_inflight{adapter}`

### 4.8 Políticas de Cache
Cache clave: hash(payload canonical) + adapter.version.
- TTL configurable (p.ej. diseño proteico 24h).
- Invalidation si cambia versión o weights.

### 4.9 Control de Recursos
Cada adapter declara `resource_profile = {"gpu":1, "mem_gb":8}`.
Scheduler valida disponibilidad antes de ejecutar.

### 4.10 Dominio Materiales (Ejemplo Fase 3)
Cadena inicial: `GNOMEHeuristicsAdapter` → `M3GNetPlaceholderAdapter` → `PropertyAggregationAdapter`.

### 4.11 NLP Biomédico
Funciones: extracción de entidades, evidencia literaria, simulación de outcomes clínicos sintéticos.
Métricas extra: `evidence_count`, `source_diversity`.

### 4.12 Testing Estratégico
- Fixtures de payloads mínimos (secuencias cortas, estructuras pequeñas).
- Mocks de adapters pesados para CI rápido.
- Golden outputs versionados para detectar drift.

### 4.13 Roadmap Fallbacks

## 5. Algoritmo de Validación Cruzada

### 5.1 Objetivo
Cuantificar coherencia multi‑dominio, detectar conflictos y producir un `consensus_score` para decisiones de refinamiento/publicación.

### 5.2 Entradas
- Métricas por dominio: `domain_scores = {domain: {evidence_score, plausibility, risk_local, confidence}}`.
- Relaciones grafo (edge weights de soporte o conflicto previos).
- Reglas duras (bioseguridad, ética) → flags.

### 5.3 Construcción Matriz C
Para dominios ordenados `D = [d1..dn]`:
```
C[i,j] = {
  compatibility: f(domain_scores[di], domain_scores[dj], edges(di,dj)),
  rationale: str,
  updated_at: ts
}
```
Propiedades:
- `compatibility ∈ [-1,1]` (simetría aproximada: |C[i,j]-C[j,i]| < ε).
- Diagonal reservada para `self_consistency` = función de varianza interna de outputs (si aplica).

### 5.4 Función de Compatibilidad (MVP)
```
compat(di,dj) = w1 * min(confidence_i, confidence_j) * sign(support - conflict) * log(1+evidence_intersection)
                - w2 * risk_overlap
```
Donde:
- `support` = suma pesos edges tipo `supports` entre nodos resultados de dominios.
- `conflict` = suma pesos edges tipo `conflicts_with`.
- `risk_overlap` = media de componentes riesgo coincidentes.
- `evidence_intersection` = |set(evidencias_i) ∩ set(evidencias_j)|.

Parámetros iniciales: `w1=0.6`, `w2=0.4`, `ε=0.05`.

### 5.5 Score Global
```
domain_aggregate = Σ_d (α_d * evidence_score_d * confidence_d) - β * Σ_d risk_local_d
compat_term = γ * ( Σ_{i<j} max(0, C[i,j].compatibility) - δ * Σ_{i<j} max(0, -C[i,j].compatibility) )
consensus_score = normalize(domain_aggregate + compat_term - hard_penalties)
```
- `α_d` normalizado según importancia declarada de dominio (peso configurable).
- `hard_penalties` = ∞ si regla dura violada → forzar invalidación.

### 5.6 Actualización Incremental
Al completarse validación de un dominio nuevo `dk`:
1. Calcular filas/columnas `C[k,*]` usando datos existentes.
2. Recalcular solo `compat_term` incremental.
3. Actualizar `metrics_snapshot.consensus`.

### 5.7 Detección de Conflictos Críticos
Condiciones:
- Si ∃ par (i,j) con `compatibility < -0.7` y ambos `confidence > 0.6` → marcar `conflict_cluster`.
- Disparar evento `hypothesis.refine.requested`.

### 5.8 Refinamiento Automático (Hook)
Estrategias:
- Solicitar evidencia adicional (adapter de literatura) centrada en variables conflictivas.
- Ajustar pesos `α_d` reduciendo impacto de dominios con alta varianza.

### 5.9 Serialización
Guardar en `cross_validation_matrix` como:
```
{
  "domains": ["protein","genomics",...],
  "matrix": [[{compatibility:0.9,...}, ...], ...],
  "params": {w1:0.6,w2:0.4,γ:0.5,δ:0.7,β:0.3},
  "version": "cv_matrix.v1"
}
```
Hash independiente para cambios estructurales.

### 5.10 Testing
- Prueba de simetría aproximada.
- Prueba de penalización conflicto (compatibility negativa reduce score global).
- Caso borde: un solo dominio → compat_term = 0.

### 5.11 Futuras Extensiones
- Inferencia bayesiana de coherencia (grafo probabilístico).
- Ajuste de pesos por aprendizaje (optimización sobre histórico de hipótesis exitosas).

- `view_entity_degrees` (ayuda a detectar hubs anómalos → posible sesgo).

FTS5: Tabla virtual `knowledge_search(content TEXT)` alimentada con concatenación de labels + propiedades textuales.

### 6.6 Embeddings
- Generar embeddings sólo para nodos textuales grandes (`Hypothesis`, `Publication`).
- Almacenamiento inicial: archivo FAISS (`faiss_index.bin`) + mapping `node_id -> vector_id`.
- Actualización batch nocturna (cron interno) para evitar latencia en escritura.

### 6.7 API de Consulta Inicial
```
get_related(node_id, relation_type=None, depth=1)
search_text(query, top_k=10)
semantic_expand(hypothesis_id) -> sugiere nodos conexos de soporte potencial
```

### 6.8 Evidencia y Peso Dinámico
Peso de edge `supports` actualizado con fórmula:
```
new_weight = decay(old_weight, half_life_days) + k * confidence_result
```
Permite atenuar evidencia obsoleta.

### 6.9 Consistencia Ontológica
Validador periódico:
- Sin ciclos ilícitos en cadena `refines` (excepto divergencia documentada).
- Cardinalidad: una `Structure` puede referirse a una sola `Protein` primaria.

### 6.10 Migración a Grafo Dedicado (Futuro)
Criterios para Neo4j/Arango:
- > 100k nodos o consultas multi-hop > 4 con latencia > 300ms.
- Necesidad de traversals probabilísticos complejos.

### 6.11 Integración con Validación Cruzada
Edges `supports/conflicts_with` alimentan cálculo de compatibilidad (sección 5). Registrar actualizaciones post-validación.

### 6.12 Pruebas
- Inserción y recuperación multi-hop.
- Búsqueda FTS coincide términos variables.
- Peso decay correcto en edge simulado.

## 8. Módulo de Ética y Riesgo

### 8.1 Objetivo
Prevenir ejecución y publicación de hipótesis con riesgo bio/ético elevado sin controles; proporcionar transparencia y trazabilidad.

### 8.2 Taxonomía Inicial de Riesgos
| Categoría | Ejemplos | Acción |
|-----------|----------|--------|
| Bioseguridad | Patógenos, resistencia antimicrobiana | Bloqueo o revisión humana |
| Químico | Synthesis peligrosa | Marcar y reducir prioridad |
| Uso Dual | Potencial weaponization | Escalada ética |
| Datos Sensibles | PHI, genómica identificable | Anonimización obligatoria |
| Impacto Ambiental | Altas emisiones compute | Recomendación ajuste recursos |

### 8.3 Scoring
```
risk_score = Σ (category_weight * likelihood * impact)
```
Normalizado 0-1. Umbrales:
- `>=0.7` → bloquear.
- `0.5-0.7` → require refine / gating manual.

### 8.4 Pipeline de Evaluación
1. Clasificación semántica (LLM + listas control) sobre `title/description`.
2. Reglas regex/heurísticas (e.g. secuencias patógenos conocidos hash).
3. Cálculo de consumo energético estimado (GPU hours * factor CO2) → `environmental_subscore`.
4. Agregación + decisión.

### 8.5 Registro en `safety_assessment`
```
{
  risk_score: 0.42,
  categories: [{name:"Bioseguridad", score:0.1}, ...],
  decisions: [{gate:"execution", allow:true, rationale:"below threshold"}],
  updated_at: ts
}
```

### 8.6 Reglas Duras (Fail Fast)
- Matching secuencia en lista negra.
- Solicitud explícita de síntesis patógena.
- Intento deanonymizar dataset.

### 8.7 Auditoría
Cada bloqueo genera evento `risk.gate.triggered` + entrada en `validation_history`.

### 8.8 Testing
- Caso: título inocuo → riesgo bajo.
- Caso: término prohibido → bloqueo inmediato.
- Caso: alta energía → subscore ambiental elevado.

| Riesgo | Descripción | Mitigación | Señal Temprana |
|--------|-------------|------------|----------------|

| Errores Silenciosos | Adapters devuelven parcial sin fallar | Contract tests y checksums | Discrepancia hash |
| Colisión Hash | Riesgo mínimo pero crítico | BLAKE3 + length prefix + verificación cruzada | Detección duplicados improbables |
